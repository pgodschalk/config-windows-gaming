# UEFI settings

Used on UEFI firmware version 7D70v1R (AGESA ComboAM5PI 1.3.0.1)

Values marked with \* are non-default.

#### `PCIe Subsystem Settings`

| Key                          | Value         | Explanation                                                                                                                                                                                                                                                                                                                                                                                              |
| ---------------------------- | ------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `Re-Size BAR Support`        | `Enabled`     | Lets the CPU address the GPU's full VRAM in a single PCIe transaction instead of 256 MB windows. Requires UEFI + above-4G decoding + supported GPU/CPU (4090 and 7800X3D both support it). Real-world gains: 1-10% FPS in supported games, no downside. Sometimes branded "Smart Access Memory" by AMD.                                                                                                  |
| `M2_1 Gen Mode`              | `Gen5` \*     | CPU-connected slots, Gen5 capable. M2_1 hosts the Crucial T700 (Gen5 NVMe). ~1-2 seconds faster POST (no negotiation handshake), eliminates the rare negotiation glitch that can drop a link to a lower speed.                                                                                                                                                                                           |
| `M2_2 Gen Mode`              | `Gen5` \*     | CPU-connected slots, Gen5 capable. ~1-2 seconds faster POST (no negotiation handshake), eliminates the rare negotiation glitch that can drop a link to a lower speed.                                                                                                                                                                                                                                    |
| `M2_3 Gen Mode`              | `Gen4` \*     | Chipset-connected slots, Gen4 max (chipset uplink itself is Gen4). ~1-2 seconds faster POST (no negotiation handshake), eliminates the rare negotiation glitch that can drop a link to a lower speed.                                                                                                                                                                                                    |
| `M2_4 Gen Mode`              | `Gen4` \*     | Chipset-connected slots, Gen4 max (chipset uplink itself is Gen4). ~1-2 seconds faster POST (no negotiation handshake), eliminates the rare negotiation glitch that can drop a link to a lower speed.                                                                                                                                                                                                    |
| `PCI_E1 Gen Switch`          | `Gen4` \*     | Chipset-connected slots, Gen4 max (chipset uplink itself is Gen4). ~1-2 seconds faster POST (no negotiation handshake), eliminates the rare negotiation glitch that can drop a link to a lower speed.                                                                                                                                                                                                    |
| `Chipset Gen Switch`         | `Gen4` \*     | Chipset-connected slots, Gen4 max (chipset uplink itself is Gen4). ~1-2 seconds faster POST (no negotiation handshake), eliminates the rare negotiation glitch that can drop a link to a lower speed.                                                                                                                                                                                                    |
| `PCI_E1 Lanes Configuration` | `Auto`        | Controls bifurcation of the CPU's x16 PCIe lanes feeding `PCI_E1`. With a single GPU in the slot (no secondary device sharing lanes), `Auto` resolves to x16 full-width.                                                                                                                                                                                                                                 |
| `ASPM Control for CPU PCIe`  | `Disabled` \* | ASPM (Active State Power Management) lets PCIe links enter low-power L0s/L1 states when idle. Cost is recovery latency every time traffic resumes (50-200µs per wake), which adds frame-time jitter in games and stutters in real-time workloads. The few watts saved at idle aren't worth the latency cost on a desktop. Affects CPU-direct PCIe links: the GPU in PCI_E1 and the M2_1/M2_2 NVMe slots. |

#### `USB Configuration`

| Key                  | Value     | Explanation                                                                                                                                                                                                                                                                                                                                                                                                       |
| -------------------- | --------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `XHCI Hand-off`      | `Enabled` | XHCI (USB 3.x) host controller handoff lets the OS take ownership of USB controllers from the UEFI during boot. Required for modern OS USB driver behavior. #### `Power Management Setup`

| Key                             | Value        | Explanation                                                                                                                                                                                                                                                                                                                                                                            |
| ------------------------------- | ------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `ErP Ready`                     | `Disabled`   | Allows full standby functionality (wake-on-LAN, wake-on-USB, USB charging while powered off, keyboard/mouse wake). ErP compliance is an EU energy regulation that caps S5/G3 standby power to <0.5W, but it does so by cutting power to USB ports and disabling all wake mechanisms. We want the convenience of those features and don't need EU eco-compliance behavior on a desktop. |
| `Restore after AC Power Loss`   | `Power off`  | System stays off after power returns from an outage instead of auto-booting. Prevents the PC from spontaneously starting in the middle of the night after a brownout or breaker trip. Also avoids POST-without-supervision during unstable power events where the PSU may cycle multiple times.                                                                                        |
| `System Power Fault Protection` | `Enabled` \* | Refuses to POST when the PSU reports a power-good fault (voltage out of spec, rail unstable, OCP triggered). Prevents booting on degraded power that could corrupt data or damage components. If the PSU ever develops an issue, you'd rather it refuse to start than spin up the system on bad rails. Free safety net with no downside in normal operation.                           |

| Key                | Value   | Explanation                                                                                                                                                                                                                                                                                                                                       |
| ------------------ | ------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `Wake Up Event By` | `OS` \* | Wake events are managed by Windows rather than the UEFI. Windows controls which devices can wake the system (configured per-device in Device Manager → Power Management tabs), timers, and scheduled tasks. #### `Advanced: AMD Overclocking`

Note: Most settings in the `AMD Overclocking` menu duplicate those in the `OC`
menu below — same AGESA options with different labels. The exception is the
Nitro signal-timing controls (`Nitro RX Data`, `Nitro TX Data`,
`Nitro Control Line`, and the related `Nitro Mode`, `Robust Training Mode`, and
Burst Length / DFE Vref entries), which are exclusive to this menu and not
surfaced in MSI's OC frontend. We configure **only** the Nitro entries here. All
other CPU, DRAM, and voltage tuning lives in `OC` where labels are clearer and
the menu hierarchy is more discoverable. If a setting appears in both menus,
treat the `OC` menu version as authoritative.

##### `DDR and Infinity Fabric Frequency/Timings`

###### `DDR Options`

| Key                            | Value       | Explanation                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| ------------------------------ | ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `DDR5 Nitro Mode`              | `Enable`    | Sets the foundation for the Nitro RX/TX/Control Line values we configured. DDR5 Nitro Mode is an MSI UEFI feature that enables aggressive memory training behavior and exposes the fine-grained Nitro signal-timing controls. When enabled, the UEFI uses more iterative training passes during POST, finding cleaner signal margins for the Nitro values to operate within.                                                                                                                            |
| `DDR5 Robust Training Mode`    | `Enable` \* | Runs more thorough DDR5 timing verification during POST. The UEFI makes additional passes to verify each timing across temperature/voltage corners before declaring training successful. Trades 2-5 seconds of additional boot time for better cold-boot reliability at tight timings.                                                                                                                                                                                                                  |
| `Nitro RX Data`                | `1` \*      | Tightens the read-side data signal sampling window (RAM → CPU memory controller direction). Lower values reduce noise/jitter margin but improve read latency by ~0.1-0.2 ns when stable.                                                                                                                                                                                                                                                                                                                |
| `Nitro TX Data`                | `2` \*      | Tightens the write-side data signal sampling window (CPU → RAM direction).                                                                                                                                                                                                                                                                                                                                                                                                                              |
| `Nitro Control Line`           | `0` \*      | Tightest setting for command/address signal timing margin (CMD/ADDR lines that tell the DRAM what to do: ACT, READ, WRITE, PRE, REFRESH). Control lines are simpler than data lines (much lower bandwidth, no high-speed bursting), so they tolerate the most aggressive value.                                                                                                                                                                                                                         |
| `Nitro Rx Burst Length`        | `Auto`      | Controls the burst length for memory read operations. DDR5's standard burst is BL16 (16 beats per access, the JEDEC default for the architecture). The IMC and DRAM are optimized around BL16; setting it manually (BC8/burst chop) rarely helps on consumer Zen 4 and can hurt bandwidth.                                                                                                                                                                                                              |
| `Nitro Tx Burst Length`        | `Auto`      | Counterpart to Nitro Rx Burst Length for write operations. Same rationale: DDR5's native burst is BL16, the IMC is built around it.                                                                                                                                                                                                                                                                                                                                                                     |
| `Nitro DFE Vref Offset Limits` | `Auto`      | Controls how far the UEFI's DFE (Decision Feedback Equalization) training is allowed to shift the reference voltage from nominal when finding optimal signal sampling points. DFE is a signal integrity technique that uses previous-bit information to clean up current-bit detection at high data rates. Wider Vref offset limits give the training algorithm more search range but cost POST time. At 6000 MT/s 1:1, DDR5 is not under enough signal stress for manual Vref Offset tuning to matter. |

### `Security`

| Key                      | Value           | Explanation                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ------------------------ | --------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `Administrator Password` | `Installed`     | UEFI administrator password required to enter setup and modify settings. Prevents anyone with physical access to the machine from changing boot order, disabling Secure Boot, resetting CMOS to defaults, or otherwise tampering with the security stack we've built up (fTPM, Secure Boot, MS Security Level 2).                                                                                                                               |
| `User Password`          | `Not installed` | User Password would gate booting (system prompts for password before POST completes). Not configured because Windows already handles user authentication (Windows Hello, account password, BitLocker if enabled) and adding a pre-boot password just creates an extra step without meaningful security benefit. The Administrator Password protects UEFI settings; Windows protects user data. No need for a third layer at the firmware level. |
| `Password Check`         | `Setup`         | Password is required only when entering UEFI setup, not at every boot. The alternative `Boot` would prompt for the password at POST on every startup — annoying for daily use and adds zero security since Windows handles user authentication after boot.                                                                                                                                                                                      |
| `Password Protection`    | `Normal`        | Stricter enforcement of the Administrator Password. Prevents common bypass methods (CMOS clear button, jumper reset, battery pull) from also wiping the password. Without Enforce, anyone with physical case access could clear the UEFI password in seconds by removing the CMOS battery for 30 seconds. With Enforce, the password persists through reset attempts and requires the actual password to disable.                               |
| `U-Key`                  | `Disabled`      | MSI-specific feature that allows a configured USB drive to act as a hardware key for UEFI access. Not used.                                                                                                                                                                                                                                                                                                                                     |

#### `Trusted Computing`

| Key                              | Value      | Explanation                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| -------------------------------- | ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `Security Device Support`        | `Enabled`  | Master toggle for TPM (Trusted Platform Module) functionality at the UEFI level. Required for Windows 11 (which mandates TPM 2.0), BitLocker disk encryption, Secure Boot's measured boot chain, and the kernel-anti-cheat compatibility stack. Specifically enables the AMD fTPM (firmware-based TPM 2.0) on Zen 4. Without this, every higher-level security feature below collapses.                                                                                                                |
| `AMD fTPM Switch`                | `Enabled`  | Activates the firmware-based TPM 2.0 implemented inside the Zen 4 SoC. Provides a Microsoft-spec-compliant TPM without needing a discrete TPM module. Required for Windows 11 install, BitLocker key storage, Windows Hello, and Microsoft Security Level 2. Worth noting: very early AGESA versions had an fTPM stuttering bug (random ~200ms freezes during fTPM access). Long since fixed in current UEFI; not a concern on any modern build.                                                       |
| `SHA256 PCR Bank`                | `Enabled`  | Activates the SHA-256 hash bank in the TPM's Platform Configuration Registers (PCRs). PCRs cryptographically record each component in the boot chain (firmware, bootloader, kernel modules) for measured boot and remote attestation. SHA-256 is the standard algorithm used by BitLocker key sealing, Microsoft Security Level 2 attestation, Windows Hello, and most security tools. Required for those features. Free to leave on.                                                                  |
| `SHA384 PCR Bank`                | `Disabled` | Activates the SHA-384 hash bank alongside SHA-256. SHA-384 is the stronger alternative (longer hash, more collision-resistant) increasingly used by enterprise security tools and forward-looking attestation schemes. Some modern Microsoft attestation paths prefer SHA-384 when available. No conflict with SHA-256 having both enabled simultaneously — TPM 2.0 supports multiple PCR banks at once. Trivial cost (a few extra TPM cycles during boot measurement), small future-proofing benefit. |
| `Pending operation`              | `None`     | Placeholder field for queuing TPM operations that execute on next reboot (clear TPM, change TPM state, etc.). `None` means no pending operation is queued. You'd only set this temporarily if intentionally wiping the TPM (which would invalidate BitLocker, Windows Hello, and any stored credentials — destructive operation). Normal state is always `None`.                                                                                                                                       |
| `Platform Hierarchy`             | `Enabled`  | Activates the platform hierarchy, the UEFI-controlled domain within TPM 2.0. Used for platform-level measurements (recording boot chain hashes into PCRs) and platform attestation. Required for Secure Boot's measured boot to function and for Microsoft Security Level 2 attestation to work end-to-end.                                                                                                                                                                                            |
| `Storage Hierarchy`              | `Enabled`  | Activates the TPM 2.0 storage hierarchy, the domain used for sealing/wrapping cryptographic keys to platform state. This is where BitLocker stores its volume encryption keys, where Windows Hello stores credential blobs, and where applications can request TPM-sealed storage (keys that only decrypt if the platform state matches what was recorded). Required for BitLocker and Windows Hello to function.                                                                                      |
| `Endorsement Hierarchy`          | `Enabled`  | Activates the TPM 2.0 endorsement hierarchy, the domain rooted in the manufacturer-provisioned Endorsement Key (EK). It proves the TPM is genuine and underpins attestation — deriving Attestation Identity Keys and supporting Windows device health attestation and Microsoft Security Level 2. Enabled is the default; required for attestation to work end-to-end.                                                                                                                                 |
| `Physical Presence Spec Version` | `1.3`      | TCG (Trusted Computing Group) Physical Presence Interface specification version. Defines the protocol the OS uses to request TPM operations that require physical user presence (clearing the TPM, changing ownership, taking control). Version 1.3 is the current standard expected by Windows 10/11 and modern security tooling. Older versions (1.2, 1.0) work but lack some features used by current Windows TPM management.                                                                       |
| `Disable Block Sid`              | `Disabled` | Confusing double-negative naming. With this set to `Disabled`, the UEFI does send the Block SID command to OPAL-capable drives at boot (Block SID is active). This prevents an attacker with physical access from taking ownership of self-encrypting drive (SED) encryption credentials.                                                                                                                                                                                                              |

#### `Secure Boot`

| Key                | Value      | Explanation                                                                                                                                                                                                                                                                                                         |

###### `Precision Boost Overdrive`: `Curve Shaper`

Curve Shaper is the second-order refinement layer on top of Curve Optimizer:
where per-core CO shifts a core's whole V/F curve by a single offset, Curve
Shaper adjusts the offset independently at 15 points — five frequency bands
(Min/Low/Med/High/Max) crossed with three temperature bins (Low/Med/High). It
lets you ease off the undervolt only in a specific corner (e.g. the cold/idle
corner where X3D CO instability surfaces) while keeping an aggressive offset
elsewhere. **All 15 points are left on `Auto`.** Per-core Curve Optimizer
already delivers the undervolt and the config is finalized and stable, so this
layer is intentionally untouched; each row below records the V/F corner that
point governs.

| `Min Frequency - Low Temperature`   | `Auto` | Offset at the lowest boost clocks with cold silicon — the idle/desktop corner where over-aggressive global CO most often triggers `Kernel-Power 41` reboots. |

| `Min Frequency - High Temperature`  | `Auto` | Offset at the lowest boost clocks when hot — an uncommon pairing, since low clocks rarely coincide with high temperatures.                                   |
| `Low Frequency - Low Temperature`   | `Auto` | Offset at low clocks with cold silicon (light load, cool).                                                                                                   |
| `Low Frequency - Med Temperature`   | `Auto` | Offset at low clocks at moderate temperatures.                                                                                                               |

Manual overclocking (e.g., 101-102 MHz) was historically used to extract marginal performance on older platforms by ramping all derived clocks proportionally. Our OC headroom comes from CPU Curve Optimizer and memory timings, not BCLK manipulation. Auto = stock 100 MHz. |
